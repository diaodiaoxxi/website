# 严格模式 vs 非严格模式

严格模式对正常的JavaScript语义上做了一些更改

  • 通过严格模式抛出的错误来消除一些原有的静默错误
  • 严格模式修复了一些导致 JavaScript引擎难以执行优化的缺陷：有时候，相同的代码，严格模式可以比非严格模式下运行得更快
  • 严格模式禁用了在ECMAScript的未来版本中可能会定义的一些语法

# 变量

  • 创建变量 严格模式下，未声明的变量直接引用会报错

  • delete删除变量名，会报错 严格模式下，不允许delete删除变量名会报错 常规模式下，也不允许delete删除变量名，但是不会报错，静默错误

  • 变量名设置 严格模式下，变量名会不允许设置与关键词同名

# 对象

非严格模式下会静默失败的情形，在严格模式下就会抛出错误。因此，在开发中使用严格模式会加大早发现错误的可能性

  • 给只读属性writable:false赋值，报 TypeError

  • 对可配置属性configuable:false，使用delete操作符，也会报 TypeError

  • 对象是不可以扩展的Object.isExtensible()，若针对对象扩展，也会报 TypeError

  • 对象的属性也必须是唯一的，不能出现重复的属性名

# 函数

  • 命名参数的参数名必须是唯一的，不能出现类似 function A(num, num) 参数名一样的情况，否则会报错

  • 函数体中的arguments 非严格模式：函数体内修改参数的值，会同步改变arguments对象的属性值 严格模式：函数体内修改参数的值，arguments对象的属性值不会改变

  • if 语句声明函数 非严格模式：将函数提升到 if 语句外部 严格模式：直接报错 语法错误

# this

非严格模式下，使用apply、call方法时，null或者undefined会(静默提升)被转化为全局对象windows 严格模式下，函数的this，始终有确定的值，无论指定的是什么值

xxx.call(null)

# 创建eval作用域、eval中使用arguments

正常模式下，js有两种变量作用域，全局作用域和局部作用域， 正常模式下eval语句作用域取决于它处于全局作用域还是函数作用域， 严格模式下eval语句本身就是作用域，不能够生成全局变量，所生成的变量只能用于eval内部
