# 任务调度系统

常见的任务：JavaScript 执行、DOM、CSS、样式布局、v8垃圾回收、用户的页面交互（鼠标点击、滚动、移动等）、浏览器resize、微任务（MutationObserver、Promise）、文件读写、WebSocket

  • 第一次迭代：引入了 高优先级 任务队列
  • 第二次迭代：针对不同的消息类型实现消息列表（优先级排序）
    • 用户交互、合成页面、默认、空闲
  • 第三次迭代：动态调度策略
    • 基于不同的场景来动态调整消息队列的优先级
    • 根据浏览器处于不同的场景，动态调整优先级的顺序
  • 任务饿死
    • Chromium 为了解决任务饿死的问题，给每个队列设置了执行权重，也就是如果连续执行了一定个数的高优先级的任务，那么中间会执行一次低优先级的任务，这样就缓解了任务饿死的情况

针对不同场景下的不同消息类型实现了不同的优先级排序的消息列表，并且针对高优先级的消息长时间执行，中间会执行一次低优先级的任务，防止任务饿死的情况

# 显示

  • 浏览器生成图像，经过GPU,发送给 显卡的后缓冲区，
  • GPU 会将后缓冲区和前缓冲区互换位置，也就是前缓冲区变成了后缓冲区，后缓冲区变成了前缓冲区
  • 显卡会 根据60HZ的 刷新率 读取 前缓冲区

# 存在问题

  • 渲染进程生成图片的 帧速 > 显卡的 刷新率，则会出现 页面丢帧 的问题
  • 渲染进程生成图片的 帧速 < 显卡的 刷新率，则会出现 页面卡顿 的问题
  • 渲染进程生成图片的 帧速 == 显卡的 刷新率，其 浏览器生成帧的时钟周期、VSync时钟周期也很难同步，则会出现 页面不连贯 的问题

# 解决

即：同步 VSync

当显示器将一帧画面绘制完成后，并在准备读取下一帧之前，显示器会发出一个垂直同步信号（vertical synchronization）给 GPU，简称 VSync

具体地讲，当 GPU 接收到 VSync 信号后，会将 VSync 信号同步给浏览器进程，浏览器进程再将其同步到对应的渲染进程，渲染进程接收到 VSync 信号之后

# 有了setTimeOut，为什么还要使用rAF（requestAnimationFrame`）？

名词：浏览器生成帧的时钟周期、VSync时钟周期

  • window.requestAnimationFrame：讲 浏览器生成帧的时钟周期 VSync时钟周期（显示器发GPU的垂直同步信号）保持一致

  • requestAnimationFrame：回调任务会在每一帧的开始执行，相当于 显卡图像更新后，就会触发该函数的调用

CSS 动画是由渲染进程自动处理的，所以渲染进程会让 CSS 渲染每帧动画的过程与 VSync 的时钟保持一致, 这样就能保证 CSS 动画的高效率执行

但是 用JavaScript来实现动画的话，需要由用户控制的，如果采用 setTimeout 来触发动画每帧的绘制，（很难 保证浏览器生成帧的时钟周期、VSync时钟周期的一致性），那么其绘制时机是很难和 VSync 时钟保持一致的，所以 JavaScript 中又引入了 window.requestAnimationFrame，用来和 VSync 的时钟周期同步

举个例子：

width宽度有100到400的过渡动画

  • 如果不添加任何操作，浏览器直接从100调到400，则没有过度的效果，图像值更新一次
  • 如果通过 setTimeout 来实现过度的效果，很难实现 VSync的时钟周期 和 浏览器生成帧的时钟周期 保持一致，就可能出现 动画不连贯 问题
  • 使用 window.requestAnimationFrame，可以保证显卡在每次更新完一帧图像后，就触发回调，这样就能保证 VSync的时钟周期 和 浏览器生成帧的时钟周期 的一致，从而达到流畅的动画

# 微任务、宏任务

V8 的宿主维护了消息队列，主线程不断从消息队列中取出任务，并依次执行，我们把消息队列中的任务称为宏任务

微任务是一个需要异步执行的函数，可以通过 Promise.reject 或者 Promise.resolve 来触发，执行时机是在当前调用 reject 或者 resolve 函数执行结束之后、当前宏任务结束之前

V8 在每个宏任务中维护了一个微任务队列
