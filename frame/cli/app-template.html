# 应用模版

默认地，工具只提供了三种基本的应用模版，但支持开发者进行扩展。

一个自定义的应用模版是一个 npm 包，用户将模板集成到本地之后，在项目创建阶段会被 HUI CLI 自动发现，具体如下：

? 请选择要创建的项目类型: (Use arrow keys)
❯ app ------------- 基础业务应用模版, 支持独立运行和部署
  frame ----------- 基础框架应用模版, 支持独立运行和部署，支持多系统集成
  pro ------------- HUI 1.0 应用模版（遗留）
  layout ---------- HUI 2.0 对接操作员中心框架应用模版（自定义）

# 模版开发

在框架 2.0 中，一个自定义的应用模版包应该具有以下的功能：

  • 渲染模版文件 - 有时需要依赖一些模版参数，这是一个基础的功能；
  • 扩展 package.json - 为你的模版指定一些必须的依赖，你需要将它们添加到 dependencies 部分时，这是一个有用的选项；
  • 提示用户选择一个特定的选项 - 例如，你可以询问用户是否集成一个自定义的组件。

模版包应该总是包含一个 Generator 为主的导出，并且能够选择性的包含 prompt 对话文件。

作为一个 npm 包，还必须有一个 package.json 文件，同时必须包含 name 和 description 字段，工具在解析模版的时候会读取这两个字段。

所以，一个自定义的模版目录结构看起来像下面这样：

.
├── README.md
├── template      # 模版文件目录（可选）
├── index.js      # Generator
├── package.json
├── prompts.js    # prompt 对话文件（可选）

# 命名和可发现性

为了让一个自定义的应用模版在 HUI CLI 项目中被正常使用，它必须遵循 @hui/boilerplates- 这样的命名惯例。这样你的插件才能够：

  • 被 @hui/boilerplates 模版服务发现；
  • 通过 hui boilerplate -a安装。

警告

确保模版的名字是正确的，否则它将无法被发现和管理

为了能够在项目创建阶段正常的显示和列举你的模版，你应该始终保证 package.json 文件的 name 和 description 字段存在并且是有意义的。

例如：

{
  "name": "@hui/boilerplates-layout",
  "version": "0.0.8",
  "description": "HUI 2.0 对接操作员中心框架应用模版"
}

# Generator

模版包应该始终保证在 index.js 中存在 Generator 为主的到处，它通常在你想要为项目扩展包依赖，渲染对应的模版文件时需要。

在用户选择对应的模版进行项目创建时，generator 会被调用。一个 generator 应该始终包含一个接收 api 作为参数的导出，具体如下：

// index.js
module.exports = function(api) {
  // api 可以用来访问模版服务对外提供的功能，比如模版渲染和依赖扩展等
};

# 模版渲染

当你调用 api.render('./template') 时，该 generator 将会使用 EJS 渲染 ./template 中的文件 (相对于 generator 中的文件路径进行解析)。

在模板文件创建完之后，你应该在 generator 中添加 api.render 调用：

// index.js
module.exports = function(api) {
  api.render("./template");
};

# 文件名的边界情况

如果你想要渲染一个以点开头的模板文件 (例如 .env)，则需要遵循一个特殊的命名约定，因为以点开头的文件会在插件发布到 npm 的时候被忽略：

# 以点开头的模板需要使用下划线取代那个点：
/template/_env

# 当调用 api.render('./template') 时，它在项目文件夹中将被渲染为：
/template/.env

同时这也意味着当你想渲染以下划线开头的文件时，同样需要遵循一个特殊的命名约定：

# 这种模板需要使用两个下划线来取代单个下划线：
/template/__variables.scss

# 当调用 api.render('./template') 时，它在项目文件夹中将被渲染为：
/template/_variable.scss

# 扩展包

如果你需要向项目中添加  额外的依赖，创建一个 npm 脚本或者修改 package.json 的其他任何一处，你可以使用 API extendPackage 方法。

// index.js
module.exports = (api) => {
  api.extendPackage({
    dependencies: {
      "hui-core": "1.0.0-beta.7",
    },
  });
};

在上面这个例子中，我们添加了一个依赖：hui-core。在插件调用时，这个 npm 模块将被安装，这个依赖将被添加到用户项目的 package.json 文件。

同样使用这个 API 我们可以添加新的 npm 任务到项目中。为了实现这个，我们需要定义一个任务名和一个命令，这样他才能够在用户 package.json 文件的 scripts 部分运行：

// index.js
module.exports = (api) => {
  api.extendPackage({
    scripts: {
      start: "hui dev",
    },
  });
};

# 对话

对话是当基于自定义模版创建项目的过程中需要与用户交互时需要的。所有的对话逻辑都存储在 prompts.js 文件中。在模版服务内部，对话是通过 inquirer 实现。

当用户选择你的模版进行项目创建时，如果模版包的根目录包含 prompts.js，它将在调用时被使用。这个文件应该导出一个问题数组，具体如下：

// prompts.js
module.exports = [
  {
    type: "confirm",
    name: "autoRouting",
    message: "是否使用内置的路由引擎自动处理路由关系？",
    default: true,
  },
];

在 generator 中，可以通过 api.promptAnswers 访问答案对象。

对话可以有不同的类型，但是在 CLI 大多数使用的是 checkbox 和 confirm。在上面的示例中，我们利用一个 confirm 对话创建了一个有条件的模板渲染。

# 模版管理

执行 boilerplate 命令可以查看当前环境中安装的应用模版，以及对指定的应用模版进行升级。具体如下：

$ hui boilerplate -h
hui boilerplate [options] [boilerplate-name]

查看和管理当前环境的应用模版

选项：
  --help, -h     显示帮助信息  [布尔]
  --add, -a      在当前环境下安装应用模版  [字符串]
  --list, -l     查看本地安装的应用模版  [布尔]
  --remove, -r   删除本地安装的应用模版  [字符串]
  --upgrade, -u  升级本地安装的应用模版，如果不指定参数将全部升级  [字符串]

# --add, -a

在当前环境下安装并注册应用模版，支持指定版本。具体如下：

$ hui boilerplate -a layout
$ hui boilerplate -a layout@latest

# --list, -l

查看本地安装的应用模版和版本。具体如下：

$ hui boilerplate -l

# --remove, -r

删除本地安装的应用模版。具体如下：

$ hui boilerplate -r layout

# --upgrade, -u

升级本地安装的应用模版，如果不指定参数将全部升级，具体如下：

$ hui boilerplate -u # 一键升级所有的应用模版
$ hui boilerplate -u layout # 单独升级 layout 应用模版
